# Домашнее задание 01 — вариант 20

## Тема
**Система управления медицинскими записями**

Основа варианта:
- данные предметной области: **Пользователь**, **Пациент**, **Запись**;
- API: создание нового пользователя, поиск пользователя по логину, поиск пользователя по маске имени и фамилии, регистрация пациента, поиск пациента по ФИО, создание медицинской записи, добавление записи к пациенту, получение истории записей пациента, получение записи по коду.

## Краткое описание решения
Выбрана сервисная архитектура с единым веб-интерфейсом и отдельными контейнерами для управления пользователями, пациентами и медицинскими записями.

Архитектура строится сверху вниз:
1. На уровне **System Context** определены роли пользователей и внешние системы.
2. На уровне **Container** выделены отдельные контейнеры по ключевым областям ответственности.
3. На уровне **Dynamic** показан архитектурно значимый сценарий создания медицинской записи и привязки её к пациенту.

## Роли пользователей
- **Администратор** — создаёт пользователей системы и ищет их.
- **Регистратор** — регистрирует пациентов и ищет пациентов по ФИО.
- **Врач** — создаёт медицинские записи, добавляет их пациентам, просматривает историю записей и открывает запись по коду.

## Внешние системы
- **Corporate Identity Provider** — внешняя система аутентификации и управления корпоративными учётными записями.
- **Notification Service** — внешний сервис для служебных уведомлений о важных действиях в системе.

## Контейнеры
- **Web Application** — единый пользовательский интерфейс.
- **API Gateway / BFF** — единая точка входа для клиентского приложения.
- **User Management Service** — управление пользователями и их поиск.
- **Patient Registry Service** — регистрация пациентов, поиск по ФИО, хранение карточек пациентов и ссылок на записи.
- **Medical Record Service** — создание и получение медицинских записей.
- **User Database** — хранилище пользователей.
- **Patient Database** — хранилище пациентов и связей пациент-запись.
- **Medical Record Database** — хранилище медицинских записей.

## Распределение ответственности по API
### Пользователи
- Создание нового пользователя — **User Management Service**
- Поиск пользователя по логину — **User Management Service**
- Поиск пользователя по маске имени и фамилии — **User Management Service**

### Пациенты
- Регистрация пациента — **Patient Registry Service**
- Поиск пациента по ФИО — **Patient Registry Service**
- Получение истории записей пациента — **Patient Registry Service**

### Медицинские записи
- Создание медицинской записи — **Medical Record Service**
- Получение записи по коду — **Medical Record Service**
- Добавление записи к пациенту — **Patient Registry Service** с проверкой существования записи через **Medical Record Service**

## Архитектурно значимый сценарий
В dynamic-диаграмме показан сценарий **«Создание медицинской записи и добавление её пациенту»**:
1. Врач работает через веб-приложение.
2. Веб-приложение обращается в API Gateway / BFF.
3. BFF вызывает Medical Record Service для создания записи.
4. Medical Record Service сохраняет запись в Medical Record Database.
5. BFF вызывает Patient Registry Service для привязки записи к пациенту.
6. Patient Registry Service проверяет наличие записи в Medical Record Service.
7. Patient Registry Service обновляет карточку пациента и историю записей в Patient Database.
8. Patient Registry Service отправляет служебное уведомление через Notification Service.

## Технологические решения
- **Frontend**: React SPA
- **Backend**: Spring Boot REST API
- **Databases**: PostgreSQL
- **Authentication**: OpenID Connect / OAuth 2.0
- **User provisioning**: SCIM / REST
- **Service communication**: HTTPS / REST
- **Database access**: JDBC / SQL
